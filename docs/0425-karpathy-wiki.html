1-1 · 先搞清楚：不是教學，是 idea file

核心主張（One-liner）

RAG 是每次 query 重新拼湊碎片；LLM Wiki 是讓 LLM 長期維護一份會累積、會自我交叉引用、會偵測矛盾的 markdown 知識庫。人類只負責策展與提問，LLM 做所有無聊的 bookkeeping。

這份 gist 的遞迴性

它自帶一層遞迴：文件本身是 schema，它描述的也是 schema。你看完它，再把它丟給 AI，AI 就幫你生一份你的 CLAUDE.md。

為什麼要這樣讀

  • 不要逐字逐句模仿，而是理解他在切的面向：Layer vs Operation
  • 三個 Layer 回答「東西放在哪」（Raw / Wiki / Schema）
  • 三個 Operation 回答「動作怎麼做」（Ingest / Query / Lint）
  • 所有複雜行為都是這兩組基元的組合

1-2 · 三層架構：Raw / Wiki / Schema

層誰寫頻率可變性
Raw sources你收集一次性Immutable（不可改）
The WikiLLM 寫每次 IngestEvolving（持續演化）
The Schema你跟 LLM 共同偶爾Co-evolving（共同演化）

為什麼 Schema 是關鍵

第三層是整個 pattern 最高明的地方。你不直接編 wiki，你編『wiki 怎麼寫』的規則。規則對了，wiki 自己會長對。

在你的專案裡，CLAUDE.md 就是 Schema。它規定：哪些 keyword 觸發哪個 domain、log 事件怎麼記錄、斜線指令觸發什麼動作。你每次改 CLAUDE.md，都是在調整『wiki 的 DNA』。

Raw sources建技規 PDF / 消防規條文 / Revit API 官方文件
The Wikidomain/*.md（但由老師寫，不是 LLM）

1-3 · Ingest / Query / Lint 三操作細節

Op 01 — Ingest · 消化吸收

  1. 讀完整份 source（不 chunk——這是跟 RAG 最大的差別）
  2. 寫一份獨立的摘要頁
  3. 更新 entity / concept 的既有頁面
  4. 建立新的 cross-links
  5. append 一行到 log.md

對應到你專案：/lessons 和 /domain 指令就是 Ingest。你已經在做了。

Op 02 — Query · 提問合成

  1. 先看 index.md 決定要載入什麼
  2. 載入 2–5 個相關頁
  3. 跨頁推理
  4. 輸出答案
  5. 判斷是否值得沉澱回 domain / lessons（這步是重點）

對應到你專案：CLAUDE.md 裡那張「Trigger Keywords → domain 檔」表，就是 index 的實作。而且你做得更精細——關鍵字直接匹配，不靠語義判斷。

Op 03 — Lint · 健檢除錯

  1. scan frontmatter 完整性
  2. 檢查 cross-references 是否失效
  3. 偵測重複定義 / 矛盾主張
  4. 列出 TODO / placeholder
  5. 產 lint-report 給人類審閱

對應到你專案：/qaqc 的 Phase 6 Content Quality Lint + /claude-md-sync。

1-5 · index.md 與 log.md

index.md — 空間性的地圖

內容導向目錄：分類 + 一行摘要 + 指向頁面。

Karpathy 的論點讓人耳目一新——在 ~100 sources 以下的規模，一份好 index.md 可以完全取代 embedding RAG。LLM 直接讀 index，自己決定要 load 哪些頁，不過向量資料庫。

2-2 · 三個關鍵差異的推理

差異 01：你的 Wiki 是人類寫的

Karpathy 的流程是「LLM 讀 PDF → 寫 wiki」。你的流程是「老師讀法規 → 老師寫 domain」。

這不是落後，是刻意特化。台灣建技規、消防規的權威性需要人類判讀：第 45 條怎麼套用在這個 case，牽涉到執業判斷，LLM 從 PDF 萃取出錯會害死人。LLM 在你的架構裡只是『消費者』，不是『作者』。

差異 02：你多了工具執行層

他的 wiki 是純知識。你的 wiki 要驅動外部系統——Revit API、Transaction、UI thread、跨版本相容性。

所以你的 Skill 比他的 Ingest/Query/Lint 更像『編排程式』——除了知識操作，還管 WebSocket 通訊、Transaction、ExternalEventManager。這層在 Karpathy 的世界根本不存在。

差異 03：你的 Wiki 是團隊共用

Karpathy 預設『一個人 + Obsidian』，單使用者場景。

你的 domain/ 是 Claude Code / Gemini CLI / VS Code Copilot / Antigravity 共用的。『不同入口、同一目的地』是你獨有的設計。這是團隊級的設計，不是個人筆記的設計。

2-3 · 為什麼 Domain 和 Skill 要分離

你跟 Anthropic 官方模型的差異

Anthropic 官方REVIT_MCP
知識放哪Skill 內部的 references/獨立的 domain/
知識與 Skill 關係1:1（綁在 Skill 裡）N:N（多個 Skill 共用）
適合場景Skill 知識不重疊BIM 法規被多工作流共用

真實案例

domain/fire-rating-check.md（一份知識）
    ├── 被 /fire-safety-check 引用（防火時效視覺化）
    ├── 被 /building-compliance 引用（法規檢討）
    └── 被 /element-coloring 引用（依防火等級上色）

如果搬進 Skill 內部：
    fire-safety-check/references/fire-rating.md     ← 複製一
    building-compliance/references/fire-rating.md   ← 複製二
    element-coloring/references/fire-rating.md      ← 複製三
    → 法規更新要改三個地方，忘改一個就不一致

對老師的意義

你寫一份 Domain，所有引用它的 Skill 都自動具備你的專業知識。你不需要懂 Skill 怎麼寫、工具怎麼用、程式碼怎麼改。

Steal 01 · log.md 的 session 級顆粒度 — 已完成

實際落地架構（三層 log 防線）

最後的解法比原建議還完整——建立了三層機制，不只 session 級：

  1. Layer 1 · git hook（AI-agnostic）：scripts/git-hooks/post-commit 在每次 commit 自動 append 事件到 log/YYYY-MM.md。任何 AI（Claude / Gemini / 未來模型）都會觸發——因為 hook 聽 git 事件，不聽 AI。
  2. Layer 2 · CLAUDE.md Logging Protocol（跨 AI）：規定 AI 執行 /lessons、/domain、/qaqc、/review、/handoff 後必須主動 append。透過 GEMINI.md → CLAUDE.md redirect，跨 client 生效。
  3. Layer 3 · /handoff 命令（session 級）：重要 session 結束時產出 session-summary 條目，五段落強制：完成／對齊概念／pending／拒絕選項／起手式。就是原建議的「think 事件」升級版。

Steal 02 · Query → file back 常態化 — 已完成

實際落地：/handoff 命令

原建議是「在 post-action 加反思 prompt」。我們實作了更強的版本——/handoff 斜線命令，session 結束時產出結構化交接：

## /handoff — Session 交接指令

何時觸發：
- 多主題研討告一段落
- MVP 落地後
- 重大架構決策完成
- 使用者手動要求跨機器延續

產出格式：
## [YYYY-MM-DD HH:MM] session-summary | 標題
- actor: {model-id} (via {client-name})
- trigger: manual (/handoff)

### 本 session 完成（帶 commit sha）
### 對齊的概念（mental model 沉澱）
### Pending（下 session 可接續）
### 拒絕的選項（省得重議）
### 下次可接續的起手式

配套：CLAUDE.md Session Start Protocol

AI 啟動時自動讀 log/YYYY-MM.md 末尾，特別注意最新的 session-summary 條目——那是上次會話的精華濃縮，讀到即可直接延續。

解決的具體問題

跨機器 / 跨 AI session 啟動成本估算：

情境Token使用者時間
沒有 handoff~15,000（讀完全部上下文 + 使用者重講）~15 分鐘
有 session-summary~7,000~0 分鐘

Steal 04 · 「Wiki is product, chat is ephemeral」 — 已完成

從「文化」升級為「制度」

原建議是「在 CLAUDE.md 加一句 Session End Protocol」靠 AI 自律。實際落地做得更強——變成跨 AI 的硬紀律：

  1. Session Start Protocol（CLAUDE.md）：AI 啟動時必須讀 log/ 末尾，特別注意最新的 session-summary——這是上次會話的精華濃縮，讀到即可延續，不用使用者重講
  2. Logging Protocol（CLAUDE.md）：規定執行 /lessons、/domain、/qaqc、/review 後必須主動 append log，不是建議、是規範
  3. /handoff 斜線命令：重要 session 結束時呼叫，強制產出五段落 session-summary，不自由發揮
  4. 跨 AI 通用：透過 GEMINI.md / AGENTS.md 單行指向 CLAUDE.md，任何 AI Client（Claude / Gemini / Copilot / 未來模型）都自動遵守這套紀律

關鍵差異：AI-agnostic 保底層

原 gist 沒提到的一件事——靠 AI 自律永遠有洞。我們的三層 log 架構中，Layer 1 是 git hook，它聽 git 事件不聽 AI。任何 AI 只要 commit 就會 fire，即使未來 AI 模型「壞掉」或不遵守 CLAUDE.md，Layer 1 仍會記錄。

實際 CLAUDE.md 段落（節選）

## Session Start Protocol（跨 AI 通用）

AI Agent 啟動時，應讀取 log/ 目錄下最新月份檔的末尾 ~60 行
（約 20 筆事件），了解專案近期動態。

特別注意最新的 session-summary 條目——那是上次重要會話的
精華濃縮（五段落：完成／對齊概念／Pending／拒絕選項／
起手式）。讀到即可直接延續，不需要使用者重述脈絡。
